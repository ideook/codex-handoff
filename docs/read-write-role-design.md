# Read/Write Role Design

This document defines the local role split for `codex-handoff` thread payloads.

It is the design reference for separating:

- the local thread payload data that Codex is allowed to read as restore context
- the local thread payload data that the current machine is allowed to write and push

Repo-level memory placement is not defined here.
See [Repo Memory Design](repo-memory-design.md) for the current policy.

This is a local storage role split. It does not introduce a new "current"
directory, and it does not change the remote `threads/` model.

This is the current design.

## Problem

Without a clear split, one machine can:

1. write new handoff artifacts locally
2. read those same local artifacts again as if they were pulled restore context
3. feed its own newly written output back into the next summarization or resume cycle

That loop weakens the handoff model. The machine stops distinguishing between:

- context that came back from the shared remote thread store
- context it just produced locally and has not yet round-tripped through sync

The design goal is to break that loop completely.

## Decision

Use two explicit local data roots under `.codex-handoff/`:

- `synced-threads`
- `local-threads`

Keep `repo.json` at the repo-local `.codex-handoff/` root only.
It is control-plane state and must not be duplicated under either role-specific
directory.

Keep the remote repo layout thread-oriented as it is today:

- repo root files remain at `repos/<repo-slug>/...`
- thread payloads remain under `repos/<repo-slug>/threads/...`

The system should be ready to read the full pulled repo context. Thread
selection is a runtime query decision, not a storage role.

## Meaning Of Each Root

### `synced-threads`

`synced-threads` is the local restore context store.

It contains the pulled repo-scoped handoff state that Codex may read when it
needs to continue work, search evidence, extract exact records, or build a
resume pack.

This directory answers the question:

- "What restore context is safe to read on this machine right now?"

### `local-threads`

`local-threads` is the local producer store.

It contains the handoff artifacts generated by the current machine from local
Codex activity. Those files are the source for push operations, but they are
not part of the default restore read path on the same machine.

This directory answers the question:

- "What has this machine produced that is ready to push?"

## Hard Invariants

These rules are the core of the design:

1. Codex default bootstrap reads for thread payloads must use `synced-threads` only.
2. `resume`, `search`, `extract`, and default reader/status flows for thread payloads must use `synced-threads` only.
3. Watcher-driven local thread updates must be written to `local-threads` only.
4. Pushes of thread payloads must upload from `local-threads` only.
5. Pulls of thread payloads must write pulled remote data into `synced-threads` only.
6. Pull must never prune or overwrite `local-threads`.
7. Push must never require `synced-threads` to be updated first.
8. The system must not copy freshly written producer thread payloads from `local-threads` into `synced-threads` on the same machine as part of a normal push path.

If any implementation path breaks one of these rules, it violates the design.

## Local Layout

Control-plane files stay at the repo-local `.codex-handoff/` root.

Data-plane files are split by role.

```text
.codex-handoff/
  repo.json
  .env.local
  sync-state.json
  conflicts/
  synced-threads/
    latest.md
    handoff.json
    thread-index.json
    current-thread.json
    threads/
      <thread-id>.jsonl
      <thread-id>.rollout.jsonl.gz
  local-threads/
    latest.md
    handoff.json
    thread-index.json
    current-thread.json
    threads/
      <thread-id>.jsonl
      <thread-id>.rollout.jsonl.gz
```

Notes:

- `repo.json`, auth, runtime state, and conflict snapshots are repo control data, not restore/publish payloads.
- `synced-threads` and `local-threads` intentionally use the same payload file schema for thread handoff artifacts so sync and inspection logic can be reused.
- `synced-threads/repo.json` and `local-threads/repo.json` must not exist. Role-specific directories must not carry repo-state copies.
- `synced-threads/memory.md`, `synced-threads/memory-state.json`, `local-threads/memory.md`, and `local-threads/memory-state.json` must not exist. Repo memory is root-only.
- `current-thread.json` may remain for compatibility, but it does not imply a separate storage role. It is only a pointer inside a role-specific store.

## Remote Layout

The remote structure stays thread-oriented.

```text
repos/<repo-slug>/
  repo.json
  thread-index.json
  current-thread.json
  memory.md
  memory-state.json
  threads/
    <thread-id>.jsonl
    <thread-id>.rollout.jsonl.gz
```

Remote naming stays stable because the key design change is local role
separation, not a remote namespace redesign.

## Allowed Operations By Role

### Readers

These read from `synced-threads`:

- `status`
- `resume`
- `context-pack`
- `search`
- `extract`
- default bootstrap logic
- any future "continue work from synced context" flow

These must not read from `local-threads` during normal operation.

Repo memory is the exception and is defined separately in
[Repo Memory Design](repo-memory-design.md).

### Producers

These write to `local-threads`:

- watcher incremental thread export
- one-shot local export before push
- any future local reconciliation that prepares outbound handoff state

These must not write producer results directly into `synced-threads`.

### Pull Paths

These write to `synced-threads`:

- `receive`
- `sync pull`
- startup pull in the agent

Pull paths may read repo control-plane files from the root, but pulled thread
payloads belong in `synced-threads`.

Pull paths must not write `repo.json` into `synced-threads`.

### Push Paths

These upload from `local-threads`:

- watcher-triggered push
- `sync push`
- `sync now`
- initial push during setup when no remote repo state exists yet

Push paths may update root `sync-state.json`, but uploaded thread payloads must
come from `local-threads`.

Push paths must not require or create `local-threads/repo.json`.

## Reader Behavior

The system should treat `synced-threads` as a full repo-scoped restore store for
thread payloads.

That means:

- selection happens inside reader logic, using `thread-index`, `current-thread`,
  `search`, `extract`, and other query-time heuristics

In other words, storage contains the whole pulled context; reading chooses what
to use from it.

## Producer Behavior

`local-threads` is not a temporary scratch directory in the disposable sense.

It is the machine-local outbound thread payload store.

That means:

- it should survive process restarts
- it should not be wiped by pull
- it may contain staged producer state that has not yet been pushed
- it is the authoritative local source for outbound sync until push completes

## Why The Names Are Functional

The names should communicate behavior without extra metaphor:

- `synced-threads`
  - these are the synced thread payloads that the system reads
- `local-threads`
  - these are the local thread payloads that the system pushes

These names are intentionally literal. They describe the system contract rather
than the storage implementation detail.

## Explicit Non-Goals

This design does not:

- redesign the remote thread bundle schema
- introduce a new current-thread directory tree
- require only one thread to be materialized for reading
- replace thread-level evidence search with summary-only restore

The remote remains thread-based. The local change is only the read/write role
boundary.
